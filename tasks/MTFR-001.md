# 🏗️ Рефакторинг в стиле Чистой Архитектуры

## 🎯 Цель

Провести рефакторинг кодовой базы в соответствии с принципами **Чистой Архитектуры** (Clean Architecture) и **SOLID**, обеспечив высокую поддерживаемость, тестируемость и масштабируемость проекта.

---

## 📐 Ключевые требования

### 🔹 Принцип единой ответственности (SRP)
> *«У класса должна быть только одна причина для изменения»*

- Каждый класс решает **одну конкретную задачу**
- Логика разделена по функциональным границам
- Отсутствие «божественных объектов» (God Objects)

### 🔹 Компактность и фокус
- Размер класса: **до 200 строк** (идеал: 50–150)
- Методы: **до 20 строк**, выполняют одно действие
- Глубина вложенности: **не более 3 уровней**

### 🔹 Слабая связанность (Low Coupling)
- Классы взаимодействуют через **интерфейсы**, а не конкретные реализации
- Зависимости направляются **внутрь** (правило зависимости)
- Внешние детали (БД, API, файловая система) вынесены за границы бизнес-логики

### 🔹 Высокая связность (High Cohesion)
- Связанные функции и данные сгруппированы в одном модуле
- Минимизация «размазанной» логики по проекту

## 🧱 Предлагаемая структура слоёв

```
src/
└── org/example/
    ├── app/                    # Точка входа, композиция зависимостей
    │   └── App.kt
    │
    ├── domain/                 # Бизнес-логика (ядро, без зависимостей)
    │   ├── model/             # Сущности и value-objects
    │   ├── repository/        # Интерфейсы репозиториев
    │   ├── service/           # Бизнес-правила и юзкейсы
    │   └── exception/         # Доменные исключения
    │
    ├── data/                   # Реализация доступа к данным
    │   ├── repository/        # Реализации интерфейсов из domain
    │   ├── source/            # Локальные/удалённые источники данных
    │   └── mapper/            # Преобразование между слоями
    │
    ├── config/                 # Конфигурация и DI-контейнер
    │   ├── Config.kt
    │   └── Module.kt
    │
    └── ui/                     # Представление (если есть)
        ├── console/           # Консольный ввод/вывод
        └── cli/               # Обработка аргументов командной строки
```

## ♻️ Практические шаги рефакторинга

### 1️⃣ Выделение доменной модели
```kotlin
// ❌ Было: смешанная логика
class FileProcessor {
    fun process(path: String) { /* чтение + бизнес-правила + запись */ }
}

// ✅ Стало: чистая сущность
data class FileMetadata(
    val originalName: String,
    val extension: String,
    val generatedName: String
) {
    fun getNewFileName(): String = "$generatedName.$extension"
}
```

### 2️⃣ Интерфейсы вместо реализаций
```kotlin
// domain/repository/FileRepository.kt
interface FileRepository {
    fun listFiles(path: Path): List<Path>
    fun renameFile(from: Path, to: Path): Result<Unit>
}

// data/repository/FileSystemFileRepository.kt
class FileSystemFileRepository : FileRepository {
    override fun listFiles(path: Path) = path.listDirectoryEntries()
    override fun renameFile(from: Path, to: Path) = runCatching { from.moveTo(to) }
}
```

### 3️⃣ Вынос конфигурации
```kotlin
// config/ConfigLoader.kt
interface ConfigLoader {
    fun load(path: Path): Result<AppConfig>
}

class IniConfigLoader : ConfigLoader {
    // парсинг .ini-файлов
}
```

### 4️⃣ Разделение юзкейзов
```kotlin
// domain/service/FileRenameUseCase.kt
class FileRenameUseCase(
    private val repository: FileRepository,
    private val nameGenerator: NameGenerator
) {
    fun execute(folder: Path): RenameResult {
        // только бизнес-логика, без ввода/вывода
    }
}
```

## 📊 Критерии качества после рефакторинга

| Метрика | Целевое значение              |
|---------|-------------------------------|
| 📏 Размер класса | ≤ 100 строк                    |
| 🔢 Количество методов в классе | 3–7                           |
| 🔗 Зависимости класса | ≤ 3 (через интерфейсы)        |
| 🧪 Покрытие тестами | ≥ 80% для `domain/`           |
| 🔄 Цикломатическая сложность | ≤ 10 на метод                 |
| 📦 Глубина наследования | ≤ 1 (предпочтение композиции) |

## 🧪 Тестируемость как индикатор

Правильно выстроенная архитектура позволяет легко писать тесты:

```kotlin
// test/domain/FileRenameUseCaseTest.kt
class FileRenameUseCaseTest {
    private val repository = FakeFileRepository()
    private val generator = FakeNameGenerator(seed = 42)
    private val useCase = FileRenameUseCase(repository, generator)
    
    @Test
    fun `should rename files with generated names`() {
        // Arrange
        val folder = Path("/test")
        
        // Act
        val result = useCase.execute(folder)
        
        // Assert
        assertTrue(result.isSuccess)
        assertEquals(3, repository.renamedCount)
    }
}
```

> ✅ Если класс сложно протестировать изолированно — это сигнал о нарушении принципов архитектуры.

## 🚀 Ожидаемые преимущества

| До рефакторинга | После рефакторинга |
|----------------|-------------------|
| 🔴 Изменения в одной части ломают другую | 🟢 Модули изолированы, изменения локальны |
| 🔴 Сложно добавить новую фичу | 🟢 Новые функции добавляются через расширение, а не правку |
| 🔴 Тесты требуют мокания всего подряд | 🟢 Юнит-тесты пишутся быстро и читаемо |
| 🔴 Новому разработчику сложно вникнуть | 🟢 Код самодокументируется через структуру и имена |
| 🔴 Бизнес-логика перемешана с инфраструктурой | 🟢 Ядро приложения не зависит от фреймворков и БД |

## 📝 Чек-лист перед коммитом

- [ ] Каждый класс имеет **однозначное название**, отражающее его ответственность
- [ ] Нет `import` из `data/` или `ui/` в пакете `domain/`
- [ ] Все зависимости передаются через **конструктор** (DI)
- [ ] Публичные методы имеют **документацию** (KDoc)
- [ ] Добавлены **юнит-тесты** для новой/изменённой логики
- [ ] Сборка проходит: `./gradlew clean build test`
