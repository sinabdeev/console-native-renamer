**Domain-Driven Design (DDD)** — это методология разработки программного обеспечения, предложенная Эриком Эвансом в 2003 году. Её суть: сложность системы управляется через глубокое погружение в предметную область и создание гибкой, выразительной модели, которая эволюционирует вместе с бизнес-требованиями.

---

## 🔷 Стратегический дизайн (Strategic Design)
*Определяет границы, контексты и высокоуровневую архитектуру*

| Принцип | Описание |
|---------|----------|
| **Ubiquitous Language (Единый язык)** | Общий словарь для разработчиков и экспертов предметной области. Термины из бизнеса становятся именами классов, методов и модулей в коде. |
| **Bounded Context (Ограниченный контекст)** | Чёткая граница, внутри которой модель и термины имеют однозначный смысл. Один термин может означать разное в разных контекстах (напр., «Клиент» в продажах ≠ «Клиент» в поддержке). |
| **Context Mapping (Карта контекстов)** | Визуализация взаимодействий между ограниченными контекстами: *Shared Kernel*, *Customer/Supplier*, *Conformist*, *ACL*, *Open Host Service* и др. |
| **Core / Supporting / Generic Subdomain** | Классификация поддоменов: <br>• **Core** — уникальная бизнес-ценность, фокус инвестиций;<br>• **Supporting** — важна, но не дифференцирует бизнес;<br>• **Generic** — стандартные решения (аутентификация, логирование). |

## 🔷 Тактический дизайн (Tactical Design)
*Паттерны для реализации модели внутри одного Bounded Context*

### 📦 Основные строительные блоки

```
┌─────────────────────────────────────┐
│ 🧱 Entity (Сущность)                │
│ • Имеет уникальный идентификатор    │
│ • Меняется во времени, но сохраняет │
│   свою идентичность (напр., Заказ)  │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 🪙 Value Object (Объект-значение)   │
│ • Не имеет идентичности, важен только│
│   набор атрибутов (напр., Адрес,    │
│   Деньги). Неизменяем (immutable).  │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 🏭 Aggregate & Aggregate Root       │
│ • Группа связанных объектов,        │
│   управляемых как единое целое.     │
│ • Доступ только через корень (Root) │
│ • Гарантирует инварианты и консистентность |
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 🔄 Domain Event (Событие предметной │
│    области)                         │
│ • Факт, что «уже произошло» в бизнесе│
│ • Используется для интеграции,      │
│   аудита, реактивной логики         │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 🛠️ Repository (Репозиторий)        │
│ • Абстракция над хранением агрегатов│
│ • Инверсия зависимостей: домен не   │
│   знает про БД                      │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 🎯 Domain Service (Сервис домена)   │
│ • Логика, не принадлежащая ни одной │
│   сущности/объекту-значению (напр., │
│   расчёт сложного налога)           │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 🚪 Application Service              │
│ • Координирует сценарии использования│
│ • Не содержит бизнес-логики, только │
│   оркестрацию: транзакции, права,   │
│   вызов доменных объектов           │
└─────────────────────────────────────┘

┌─────────────────────────────────────┐
│ 🧭 Factory (Фабрика)                │
│ • Инкапсулирует сложную логику      │
│   создания агрегатов/сущностей      │
└─────────────────────────────────────┘
```

## 🔷 Ключевые принципы DDD

1. **Фокус на доменной логике, а не на инфраструктуре**  
   Бизнес-правила живут в доменном слое, а не в контроллерах или скриптах БД.

2. **Изоляция домена от внешних зависимостей**  
   Домен не импортирует фреймворки, БД, HTTP-библиотеки. Используются порты и адаптеры (Hexagonal Architecture).

3. **Инварианты внутри агрегатов**  
   Бизнес-ограничения проверяются и поддерживаются на уровне агрегата, а не в сервисах или БД.

4. **Модель эволюционирует через рефакторинг**  
   Единый язык и обратная связь от экспертов позволяют уточнять модель по мере изучения домена.

5. **Явные границы транзакций**  
   Одна команда → один агрегат → одна транзакция. Для распределённой консистентности — сага + события.

6. **События как способ интеграции**  
   Domain Events позволяют контекстам взаимодействовать асинхронно и слабо связно.

## 🔷 Когда применять DDD?

✅ Сложная, быстро меняющаяся предметная область  
✅ Высокие требования к гибкости и поддерживаемости  
✅ Команда готова инвестировать в обучение и совместную работу с бизнесом  
✅ Проект масштабируется по модулям/микросервисам  

❌ Простые CRUD-приложения  
✅ Жёсткие сроки и фиксированные требования  
✅ Отсутствие доступа к экспертам домена

## 🔷 Связь с современными практиками (2024–2026)

| Практика | Как сочетается с DDD |
|----------|----------------------|
| **Микросервисы** | Один Bounded Context ≈ один сервис (но не всегда!) |
| **Event Storming** | Воркшоп для быстрого выявления агрегатов, событий и контекстов |
| **CQRS** | Разделение моделей чтения/записи усиливает тактические паттерны DDD |
| **Event Sourcing** | Хранение состояния как последовательности Domain Events — естественное расширение DDD |
| **Clean/Hexagonal Architecture** | Техническая реализация изоляции домена, пропагандируемой в DDD |

---

💡 **Совет**: Не начинайте с тактических паттернов. Сначала проведите **Event Storming** или **Domain Storytelling**, чтобы выявить поддомены, контексты и единый язык. Паттерны вроде Aggregate или Repository — это инструменты, а не цель.
